# MyCar Pro — Smart Factory: симулятор линии сборки

Симулятор конвейерной линии сборки кузовов (учебный проект, контур MES / Smart Factory).

Кузова продвигаются по станциям по тактам. Любой кузов можно снять "на доработку"
и вернуть обратно на линию с заданной позицией, менять приоритет во входной очереди.
Все перемещения фиксируются в журнале событий. Симулятор считает метрики линии
(throughput, avgLeadTime, загрузку станций), находит узкое место и рисует схему линии.

> Статус: **готово к демо** — неделя 2 выполнена (метрики, узкое место, схема)

## Запуск

Требования: Python 3.10+

```bash
python3 src/main.py                              # загрузка + 14 тактов демо-сценария
python3 src/main.py 20                           # загрузка + 20 тактов
python3 src/main.py 40 config/demo_config.json   # демо-конфиг с заведомым узким местом
```

> На macOS команда называется `python3` (просто `python` может отсутствовать).

Демо-прогон загружает линию из `config/line_config.json`, печатает стартовое
состояние, затем на каждом такте:
- выполняет `tick()` и печатает состояние линии (ASCII-схема, станции, входная
  очередь, буфер доработки);
- на заранее заданных тактах выполняет операции `send_to_rework`, `change_priority`
  и `return_to_line`, чтобы наглядно показать все возможности симулятора;
- в конце печатает полный журнал событий и метрики прогона (completed,
  throughput, avgLeadTime, загрузку станций с пометкой узкого места).

Сценарий действий задаётся в `DEMO_ACTIONS` в начале `src/main.py` — можно
менять такты и id кузовов, чтобы проиграть свой сценарий.

## Веб-интерфейс

```bash
python3 src/server.py          # запуск на http://localhost:8000
python3 src/server.py 9000     # другой порт
```

Открыть http://localhost:8000 в браузере. Вся логика симуляции выполняется
Python-ядром (`src/engine.py`); страница `index.html` — только отображение
и панель управления: такт/автопрогон с выбором темпа, добавление кузова,
отправка на доработку, возврат в очередь, повышение приоритета («вне
очереди»), сброс к конфигу. Двойной клик по станции или кузову —
переименование (выполняется ядром, фиксируется в журнале); ползунок
«станции» меняет размер карточек. Сервер — стандартная библиотека Python,
дополнительные зависимости не нужны.

## Метрики

После каждого прогона `main.py` печатает метрики; веб-интерфейс показывает их
в панели «Метрики линии» и подсвечивает узкое место на доске станций.

Формулы:

| Метрика | Формула | Смысл |
| --- | --- | --- |
| `completed` | число кузовов со статусом «готов» | сколько машин выпущено за прогон |
| `throughput` | `completed / totalTicks` | пропускная способность (кузовов за такт) |
| `leadTime` кузова | `tickFinished − tickEntered` | сколько тактов кузов шёл от входа до готовности |
| `avgLeadTime` | среднее `leadTime` по готовым кузовам | среднее время прохождения линии (`null`, если готовых нет) |
| `utilization` станции | `busyTicks / totalTicks` | доля тактов, когда станция была занята (0..1) |
| bottleneck | станция с максимальной `utilization` | узкое место, ограничивающее выпуск всей линии |

Крайние случаи: `totalTicks = 0` → `throughput = 0`, узкое место не определяется
(`null`); нет готовых кузовов → `avgLeadTime = null` (без деления на ноль).
При равной загрузке узким местом считается первая станция по порядку линии.
`tickEntered` запоминается при первом входе на линию — возврат с доработки
не сбрасывает его, поэтому доработка честно удлиняет lead time.

Пример прогона демо-конфига (`python3 src/main.py 40 config/demo_config.json`,
станция «Покраска» — заведомое узкое место, `processingTicks = 6` против 1):

```
=== Метрики прогона (40 тактов) ===
  completed   = 4 (кузовов готово)
  throughput  = 0.100 (кузовов за такт)
  avgLeadTime = 18.8 (тактов от входа до готовности)

  Загрузка станций (utilization):
    Сварка кузова           0.93 |##################  |
    Покраска                0.95 |################### |  <- узкое место *
    Сборка интерьера        0.25 |#####               |
    Установка двигателя     0.23 |####                |
    Финальный контроль      0.20 |####                |
```

«Сварка» тоже загружена на 0.93 — её блокирует стоящая следом «Покраска»:
очередь копится именно перед узким местом.

## Визуальная схема линии

При каждом выводе состояния печатается однострочная схема: входная очередь →
станции с VIN стоящего кузова («—» — станция пуста, `*` — узкое место) → готовые.

```
Схема: [Вход:1] -> (Сварка кузова:KZ0006MYCAR) -> (Покраска:KZ0005MYCAR *) -> (Сборка интерьера:—) -> (Установка двигателя:—) -> (Финальный контроль:KZ0004MYCAR) -> [Готово:3]
```

43 unit-теста в `tests/`, покрывают:
- продвижение кузова по тактам с учётом времени обработки станции;
- блокировку продвижения занятой станцией;
- `send_to_rework` — снятие в буфер, немедленное освобождение станции;
- `return_to_line` — вставка в очередь на заданную позицию (в т.ч. clamp за границами);
- `change_priority` — пересортировку очереди по приоритету;
- метрики: `completed`/`throughput`/`avgLeadTime` с ручной проверкой чисел
  (3 станции, 5 кузовов), `avgLeadTime = null` без готовых кузовов;
- узкое место: заведомо медленная станция определяется как bottleneck,
  utilization в диапазоне 0..1, при равной загрузке — первая по порядку,
  `totalTicks = 0` → bottleneck = null;
- ASCII-схему линии: очередь/станции/готовые, пометка узкого места `*`;
- граничные случаи: пустая линия, линия без станций, все станции заняты,
  операции над несуществующим/неподходящим по статусу кузовом;
- загрузку реального конфига `config/line_config.json`.

## Формат конфига

Файл `config/line_config.json`:

```json
{
  "stations": [
    { "id": "st1", "name": "Сварка кузова", "order": 0, "capacity": 1, "processingTicks": 2 }
  ],
  "bodies": [
    { "id": "b1", "vin": "KZ0001MYCAR", "model": "MyCar Pro", "priority": 10 }
  ]
}
```

- `stations` — список станций линии, `order` задаёт их порядок на линии,
  `processingTicks` — сколько тактов кузов обрабатывается на станции.
- `bodies` — стартовый набор кузовов; входная очередь строится сортировкой
  по `priority` (меньше число — выше приоритет).

## API ядра (`src/engine.py`)

| Функция | Что делает |
| --- | --- |
| `tick(state)` | Продвигает линию на один такт: обработка станций от последней к первой, продвижение при готовности и свободной следующей станции, затягивание нового кузова из очереди на первую станцию |
| `send_to_rework(state, body_id)` | Снимает кузов со станции в буфер доработки, станция освобождается немедленно |
| `return_to_line(state, body_id, position)` | Возвращает кузов из буфера доработки во входную очередь на позицию `position` (значение ограничивается допустимым диапазоном) |
| `change_priority(state, body_id, new_priority)` | Меняет приоритет кузова, пересортировывает входную очередь (меньше число — выше приоритет, сортировка стабильна) |
| `add_body(state)` | Создаёт кузов с автоматическими id/VIN в конец входной очереди |
| `rename_station(state, station_id, name)` | Переименовывает станцию (пустое имя запрещено) |
| `rename_body(state, body_id, name)` | Задаёт кузову отображаемое имя; пустая строка сбрасывает к id |

Все операции пишут запись в `state.event_log`: такт, тип события, кузов, откуда → куда.

## API метрик (`src/metrics.py`)

| Функция | Что возвращает |
| --- | --- |
| `get_metrics(state, total_ticks)` | `{completed, throughput, avg_lead_time}` — метрики прогона |
| `get_station_load(state, total_ticks)` | `[{station_id, name, utilization}, ...]` — загрузка станций по порядку линии |
| `get_bottleneck(state, total_ticks)` | Станция с максимальной utilization (`None`, если `total_ticks = 0` или станций нет) |
| `format_line_schema(state, bottleneck_id)` (в `display.py`) | Однострочная ASCII-схема линии с пометкой узкого места `*` |

## Терминология (глоссарий)

- **Линия** — последовательность станций, через которые проходит кузов.
- **Станция** — точка обработки кузова (сварка, покраска и т.д.).
- **Кузов** — единица продукции, движущаяся по линии.
- **Такт** — один шаг симуляции (`tick`), за который кузова могут продвинуться.
- **FIFO** — принцип "первым пришёл — первым обслужен" для входной очереди.
- **Пропускная способность (throughput)** — сколько кузовов линия выпускает за такт.
- **Время прохождения (lead time)** — сколько тактов кузов идёт от входа до готовности.
- **Загрузка (utilization)** — доля времени, когда станция занята (0..1).
- **Узкое место (bottleneck)** — самая загруженная станция, ограничивающая всю линию.

## Пример сценария (из демо в `main.py`)

1. Такт 1 — `b1` заезжает на первую станцию.
2. Такт 3 — `b1` отправляется на доработку (`send_to_rework`), станция освобождается.
3. Такт 4 — освободившееся место сразу занимает следующий кузов из очереди.
4. Такт 5 — `b4` получает наивысший приоритет (`change_priority`) и обходит остальных в очереди.
5. Такт 7 — `b1` возвращается в начало очереди (`return_to_line`, `position=0`).
6. В конце прогона печатается полный журнал событий по всем тактам.
