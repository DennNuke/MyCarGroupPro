# День 5 — Демо и защита (5–7 минут)

Формула выступления: **проблема → решение → живая демонстрация → польза для бизнеса**.

## 1. Проблема (≈1 мин)

Линия сборки — конвейер из станций: сварка, окраска, монтаж. Каждая операция
занимает время. Если одна станция медленнее других, перед ней копится очередь,
а вся линия простаивает — это «узкое место» (bottleneck), и именно оно
ограничивает выпуск.

Вопрос: **как понять пропускную способность линии и где она «затыкается»,
не трогая реальный цех?** Эксперименты на живой линии — это простой
оборудования и деньги.

## 2. Решение (≈1 мин)

Симулятор линии на Python (контур MES):

- линия любой конфигурации поднимается из JSON-файла;
- кузова движутся по тактам (FIFO, блокировка занятой станцией, доработка,
  приоритеты) — это было готово на прошлой неделе;
- **новое на этой неделе**: метрики `throughput` / `avgLeadTime`, загрузка
  станций (`utilization`), автоматический поиск узкого места и наглядная
  схема линии (консоль + веб-интерфейс).

## 3. Живая демонстрация (≈3 мин)

### Шаг 1 — запустить линию из демо-конфига

```bash
python3 src/main.py 40 config/demo_config.json
```

В `demo_config.json` станция «Покраска» — заведомое узкое место:
`processingTicks = 6` против `1` у остальных.

### Шаг 2 — показать движение кузовов по тактам

На каждом такте печатается схема линии:

```
Схема: [Вход:2] -> (Сварка кузова:KZ0003MYCAR) -> (Покраска:KZ0002MYCAR *) -> (Сборка интерьера:—) -> (Установка двигателя:—) -> (Финальный контроль:—) -> [Готово:1]
```

Обратить внимание: перед «Покраской» кузова стоят и ждут (`готов · ждёт
перехода`), станции после неё — пустые. Очередь копится именно перед узким местом.

### Шаг 3 — снять кузов «на доработку» и вернуть

Веб-интерфейс нагляднее для этого шага:

```bash
python3 src/server.py     # http://localhost:8000
```

Кнопки «На доработку» → «Вернуть в очередь»: станция освобождается сразу,
кузов возвращается в очередь, всё фиксируется в журнале событий.

### Шаг 4 — показать метрики и узкое место

В конце консольного прогона:

```
=== Метрики прогона (40 тактов) ===
  completed   = 4 (кузовов готово)
  throughput  = 0.100 (кузовов за такт)
  avgLeadTime = 18.8 (тактов от входа до готовности)

  Загрузка станций (utilization):
    Сварка кузова           0.93 |##################  |
    Покраска                0.95 |################### |  <- узкое место *
    ...
```

В веб-интерфейсе — панель «Метрики линии»: счётчики + полосы загрузки,
узкое место подсвечено оранжевым и на самой карточке станции (`*`).

Хороший ход на защите: «а что если ускорить Покраску?» — поменять
`processingTicks` 6 → 2 в конфиге, перезапустить и показать, что throughput
вырос, а узкое место переехало на другую станцию.

## 4. Польза для бизнеса (≈1 мин)

- Инструмент оценки линии **до** вмешательства в реальный цех: любую
  конфигурацию можно «прогнать» за секунды и увидеть выпуск и узкое место.
- Метрики — те же, что в реальном MES: throughput, lead time, utilization.
- Решения об ускорении/дублировании станции принимаются по цифрам, а не
  по ощущениям.
- Следующий шаг: перенос фичи в бэклог MMS.

## Возможные вопросы и ответы

- **Почему «Сварка» тоже загружена ~0.93, если медленная — «Покраска»?**
  Её блокирует стоящая следом «Покраска»: кузов отработал, но уйти не может.
  Это и есть эффект узкого места — оно «заражает» станции выше по потоку.
- **Почему avgLeadTime считается только по готовым кузовам?**
  У незавершённых нет `tickFinished`; включать их — занижать/искажать метрику.
- **Что при totalTicks = 0 или пустой линии?**
  `throughput = 0`, `avgLeadTime = null`, bottleneck = `null` — крайние случаи
  покрыты тестами, деления на ноль нет.
- **Как проверены цифры?**
  Тест с ручным подсчётом: 3 станции, 5 кузовов, 15 тактов →
  `completed = 5`, `avgLeadTime = 6.0`, `throughput = 1/3` (43 теста зелёные).
